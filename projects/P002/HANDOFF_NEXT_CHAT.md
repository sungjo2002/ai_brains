## 1. 프로젝트 핵심 목표

P002는 단순 AI 채팅 프로그램이 아니라 `1인 기업 AI 사무실`이다.

최종 목표는 다음 구조다.

```text
성조님 출근
→ 프로그램 실행
→ 회사 목표 확인
→ 대표비서 AI 가동
→ AI 직원들이 목표 달성을 위해 순차적으로 업무 수행
→ 인터넷 자료수집 / 쇼핑몰 분석 / 상품 설명 / CS 초안 / 이미지 프롬프트 / 기억 저장
→ 상황판에 결과 표시
→ 작업 이력과 지식창고에 저장
```

기본 목표:

```text
대표가 채팅창에 지시하면 대표비서 AI가 내용을 분석하고,
필요한 AI 직원과 도구를 연결해 업무를 처리하는 1인 기업용 AI 자동화 프로그램을 만든다.
```

## 2. 성조님이 원하는 프로그램 방향

### 2.1 회사 목표를 직접 입력하는 구조

프로그램 처음 시작 또는 로고/메뉴 클릭 시 `목표 설정 팝업`이 열려야 한다.

사용자가 직접 입력해야 하는 항목:

```text
회사 이름
회사 소개
타깃 고객
주력 상품
연간 목표
이번 달 목표
오늘 목표
우선순위
금지사항 / 주의사항
```

중요 조건:

```text
예시 문구를 실제 목표처럼 고정해서 넣으면 안 된다.
예시는 placeholder로만 표시한다.
목표 입력은 업무 상황판에 고정으로 보이면 안 된다.
목표 설정은 팝업으로 열려야 한다.
회사 이름을 저장하면 상단 로고명이 회사 이름으로 바뀌어야 한다.
```

최근 목표 설정 관련 생성본:

## 3. AI 운영 구조 기준

### 3.1 AI 직원 10명은 역할이지 모델 10개 동시 실행이 아니다

AI 직원은 화면에서 역할로 표시한다.
실제 모델은 필요한 순간에만 제한적으로 실행한다.

```text
AI 직원 10명 = 역할 / 부서
실제 실행 모델 = 현재 필요한 1~2개 중심
작업 처리 = 작업 큐에 넣고 순서대로 실행
```

한 번에 모든 AI를 돌리면 PC 과부하가 올 수 있으므로 금지한다.

### 3.2 대표비서 AI 중심 구조

가장 안정적인 구조:

```text
성조님 입력
→ 대표비서 AI
→ 의도 분류
→ 업무 라우터
→ 담당 AI / 실제 도구 실행
→ 상황판 표시
→ 작업 이력 / 기억 저장 / 지식창고 저장
```

대표비서 AI는 모든 지시의 입구다.
대표비서 AI가 먼저 아래를 판단한다.

```text
일반 대화인가?
기능 질문인가?
실제 업무 명령인가?
인터넷 검색 명령인가?
이미지 업무인가?
기억 저장인가?
승인이 필요한 위험 작업인가?
```

## 4. 모델 / 엔진 운영 기준

### 4.1 LM Studio 기준

현재 성조님 PC에서는 LM Studio가 더 안정적으로 보인다.
일반 대화는 LM Studio `google/gemma-4-e2b`가 빠르고 잘 된다.
`google/gemma-4-e4b`는 느려서 일반 대화 기본값으로 쓰지 않는다.

현재 운영 기준:

```text
일반 대화 / 대표비서 빠른 응답
→ LM Studio google/gemma-4-e2b

긴 글 / 상품설명 / 문서작성 후보
→ LM Studio google/gemma-4-e4b, 필요할 때만

Ollama
→ 보조 / fallback / 특수 역할용
```

LM Studio 권장 상태:

```text
READY = google/gemma-4-e2b 1개만 유지
기본 서버 포트 = 1234
OpenAI compatible API 사용
```

최근 65~67번 흐름:

```text
65_lmstudio_e2b_default_chat_model_fix.zip
66_lmstudio_e2b_chat_timeout_fix.zip
67_lmstudio_e2b_only_general_chat_fix.zip
```

핵심 기준:

```text
일반 대화는 LM Studio e2b만 우선 사용한다.
e4b는 일반 대화 fallback에서 제외한다.
Ollama gemma4:latest / gemma4:e2b 일반 대화 fallback은 제거한다.
e2b가 없거나 실패하면 무리하게 다른 무거운 모델로 넘어가지 말고 정확한 오류를 표시한다.
```

### 4.2 Ollama 기준

Ollama에는 다음 모델이 있었다.

```text
gemma4:e4b
llama3.2-vision:latest
nomic-embed-text:latest
qwen2.5-coder:3b
llama3.2:3b
gemma4:latest
gemma4:e2b
```

분류:

```text
일반 대화 가능 후보:
gemma4:e4b
qwen2.5-coder:3b
llama3.2:3b
gemma4:latest
gemma4:e2b

전용 모델:
nomic-embed-text:latest = 지식검색 전용
llama3.2-vision:latest = 이미지분석 전용
```

Ollama는 500 오류와 timeout이 있었으므로 일반 대화 주력으로 두지 않는다.

## 6. 목표 기반 운영 구조

성조님이 원하는 핵심은 `회사 목표를 기준으로 AI 직원이 일하는 구조`다.

예시 목표는 프로그램에 고정하지 않는다.
아래는 개념 예시일 뿐이다.

```text
회사명: 성조님이 직접 입력
타깃: 성조님이 직접 입력
연간 목표: 예를 들어 연 1억 이상
이번 달 목표: 성조님이 직접 입력
오늘 목표: 성조님이 직접 입력
```

목표가 저장되면 AI들은 이 목표를 바탕으로 업무 우선순위를 정한다.

```text
대표비서 AI = 오늘 업무 브리핑
쇼핑몰 AI = 상품 / SEO / 가격 / 상세설명 개선 제안
마케팅 AI = 키워드 / 블로그 / 홍보 문구 제안
지식관리 AI = 주기적 인터넷 자료수집
기억관리 AI = 결정사항 / 피드백 저장
이미지 AI = 이미지 프롬프트 / ComfyUI 연결
```

## 7. 자가 학습 표현의 실제 구현 방식

성조님이 말한 `자가 학습`은 모델 자체를 재학습시키는 의미로 구현하지 않는다.
현실적인 구현은 아래 방식이다.

```text
기억 저장소
+ 지식창고
+ 작업 이력
+ 성조님 피드백
+ 주기적 인터넷 자료수집
```

즉:

```text
성조님 목표 저장
성조님 말투 / 기준 저장
검색 자료 저장
작업 결과 저장
성과 기록 저장
다음 답변과 업무 배정에 반영
```

## 8. 기억 저장소 / 지식창고 기준

기억 저장소는 성조님의 기준, 선호, 피드백, 프로젝트 상태, 작업 이력을 저장한다.
지식창고는 회사 문서, 인터넷 수집 자료, 상품 자료, SEO 자료 등을 저장한다.

중요 원칙:

```text
많이 저장하는 것보다 잘 분류하는 것이 중요하다.
중복 자료를 정리한다.
오래된 자료와 최신 자료를 구분한다.
중요한 기억은 우선순위를 둔다.
폐기된 기준은 deprecated 상태로 둔다.
최신 기준을 우선한다.
틀린 답변과 이상한 답변은 보정 기록으로 저장한다.
성조님의 피드백을 다음 답변 규칙에 반영한다.
```

## 9. PC 상태 기반 작업 큐 구조

성조님은 컴퓨터 상태를 모니터링하면서 AI가 자동으로 2~3개씩 돌아가고, 가벼운 작업이면 더 많이 돌고, 무거운 작업이면 줄어드는 구조를 원한다.

기준:

```text
가벼운 작업:
일반 대화, 기억 저장, 간단 분류, 로그 저장
→ 여러 개 가능

중간 작업:
인터넷 검색, 검색 결과 요약, 상품 설명 짧은 초안, 고객 답변 초안
→ 2~3개 정도

무거운 작업:
e4b 긴 글, 긴 문서 요약, 이미지 생성, 이미지 분석, ComfyUI 실행
→ 1개씩 순차 실행
```

시스템 정책 예시:

```text
CPU 50% 이하 / RAM 60% 이하
→ 작업 3개까지 실행

CPU 70% 이하 / RAM 75% 이하
→ 작업 2개까지 실행

CPU 80% 이상 또는 RAM 85% 이상
→ 새 작업 중지, 현재 작업만 유지

이미지 생성 중
→ 다른 무거운 AI 작업 중지

일반 대화 중
→ e2b 하나만 사용
```

최근 68번에서 목표 기반 운영과 작업 큐/시스템 상태 패널을 넣으려 했다.

```text
68_goal_based_resource_scheduler_fix.zip
```

이후 목표 설정 팝업 문제로 69~73번을 진행했다.

## 10. 이미지 기능 기준

현재 이미지 요청은 실제 이미지 생성이 아니라 이미지 프롬프트 초안까지만 안전 처리한다.
ComfyUI 실제 연결은 다음 단계다.

정상 구조:

```text
이미지 만들어줘
→ 이미지 업무로 분류
→ ComfyUI 연결 여부 확인
→ 미연결이면 실제 생성하지 않음
→ 이미지 프롬프트 초안 작성
→ 상황판 중앙에 표시
→ generated_images/prompts/에 저장
```

이후 단계:

```text
ComfyUI 실행 확인
ComfyUI API 연결
workflow 연결
생성 중 표시
완성 이미지 상황판 미리보기
generated_images 폴더 저장
작업 이력에 이미지 결과 저장
```
