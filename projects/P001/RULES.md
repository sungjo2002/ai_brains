# RULES

이 파일은 P001 프로젝트의 고정 규칙과 판단 기준을 기록한다.

## 파일명 규칙

- PC 수정본: `pc_번호_영문설명.zip`
- 모바일 수정본: `mobile_번호_영문설명.zip`
- 서버 수정본: `server_번호_영문설명.zip`
- PC, 모바일, 서버 수정본은 각각 따로 만든다.
- ZIP 안에는 불필요한 최상위 바깥 폴더를 만들지 않는다.
- 불필요한 캐시 파일은 포함하지 않는다.

## 수정 실행 규칙

분석 요청과 실제 수정 요청을 구분한다.

분석 요청 예시:

```text
하자
할까
정리하자
순서대로 하자
어떨까
```

실제 수정 요청 예시:

```text
수정해
적용해
파일 만들어줘
생성해
ZIP 만들어줘
저장해
저장소 정리해
```

실제 수정 전에는 먼저 `수정 대기`를 출력하고 수정 대상, 이유, 범위, 파일명, 확인 방법을 짧게 정리한다.

## 작업 저장 기준

- 모든 사적인 대화가 아니라 실제 작업 기준, 작업 기록, 완료 내용, 다음 작업만 저장한다.
- 프로그램 수정 중에는 사용자가 명확히 요청하지 않는 한 저장소를 자동 갱신하지 않는다.
- 최신 상태는 `CURRENT.md`를 기준으로 본다.
- 다음 작업은 `TASKS.md`를 기준으로 본다.
- 작업 기록은 `HISTORY.md`에 남긴다.
- 중요한 결정은 `DECISIONS.md`에 남긴다.

## 동기화 기본 원칙

- 프로그램에서 사용하는 모든 업무값과 설정값은 기본적으로 동기화 대상으로 본다.
- 새 기능을 만들 때도 값 저장은 로컬 전용이 아니라 서버 동기화와 백업 복구 대상에 포함하는 것을 기본값으로 한다.
- 동기화에서 제외할 값은 예외 항목으로 명확히 기록한 뒤 제외한다.
- PC별 환경값은 예외로 본다.
- 사업자, 근무사업장, 근로자, 근태, 차량, 급여, 월마감, 관리자와 권한, 공휴일 자동갱신 설정, 회사 운영 공용 설정값, 사업장별 시간 환산 기준은 동기화 대상이다.

## 분석 기준

- 항상 현재 상태와 지난 기준을 둘 다 확인한다.
- 현재 상태는 사용자가 방금 올린 화면, 현재 적용된 ZIP, 현재 서버/모바일/PC 동작 결과를 기준으로 본다.
- 지난 기준은 이전 정상 상태, 이전 수정본, 작업기록, 사용자가 이미 확인한 결과를 기준으로 본다.
- 오류 분석은 현재 현상만 보고 판단하지 말고, 이전 정상 기준과 비교해서 무엇이 달라졌는지 먼저 찾는다.
- 저장소 내용만 보지 않고, 실제 업로드 파일 상태와 사용자가 확인한 실행 결과를 함께 본다.
- 실제 파일과 사용자 확인 결과가 저장소 기록보다 최신이면 실제 파일과 확인 결과를 우선한다.

## UI 수정 기준

- UI 수정은 화면 배치, 여백, 크기, 색상, 버튼 위치, 표시 방식 중심으로 한다.
- 기능 조건, 저장 조건, 동기화 조건, 계산 조건, 권한 조건은 임의로 바꾸지 않는다.

## 금지

- 기능 조건, 권한 조건, 저장 조건, 동기화 조건, 계산 조건은 임의로 바꾸지 않는다.
- 서버 방어 로직과 화면 숨김을 혼동하지 않는다.
- 운영 정보나 개인 정보성 파일은 저장소에 넣지 않는다.
