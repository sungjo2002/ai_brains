## 작업 기준 정리

### 동기화 기본 방향

- PC1이 메인 PC가 아니라, 어느 PC 또는 모바일에서 등록/수정/삭제하더라도 서버를 기준으로 다른 PC와 모바일에 반영되는 구조로 잡는다.
- 사용자는 동기화 버튼을 기존처럼 하나만 사용한다.
- 내부적으로는 저장/불러오기/병합/삭제 반영을 자동 처리하되, 사용자 화면에는 복잡한 버튼을 늘리지 않는다.
- PC ↔ 모바일 동기화는 테스트 완료 기준으로 본다.
- PC ↔ PC 서버 백업/복구 동기화 테스트는 아직 별도 확인이 필요하다.

### 이미지 동기화 기준

- 서버 사진 동기화 보강에서 제외할 대상은 다음으로 확정했다.
  - 신분증 원본 이미지
  - 여권 원본 이미지
  - 신분증 보정본 이미지
  - 여권 보정본 이미지
- 근로자 얼굴 사진은 위 제외 대상과 별도이며, 실제 구현 상태는 최신 코드 기준으로 확인이 필요하다.

### PC별 환경값과 공용 업무 설정값 구분

동기화 제외 유지:

- 서버 주소
- 백업 폴더 위치
- 창 크기/위치
- 로컬 캐시
- 로컬 실패 대기열
- PC별 실행 환경
- PC별 인증서/실행 파일 관련 설정

동기화 대상:

- 사업자 설정
- 근무사업장 설정
- 근로자
- 근태
- 차량
- 급여
- 월마감
- 관리자/권한
- 공휴일 자동갱신 설정
- 공휴일 자동갱신 인증키
- 외부 API 인증키 등 회사 운영 공용 설정값

## 수정본별 작업 내용

### server_11 ~ server_13 흐름

- PC에서 서버 전체 저장/불러오기 관련 문제를 확인했다.
- 다른 PC에서 서버 전체 불러오기 실패가 발생했다.
- 삭제 동기화에서 tombstone 방식 보강이 필요하다고 판단했다.
- 삭제한 근로자 목록이 다른 PC/모바일에 남거나, 동기화 실패 건수로 표시되는 문제를 수정 대상으로 잡았다.
- 최종 서버 기준은 `server_13_employee_delete_tombstone_fix.zip`로 정리한다.

### pc_29_employee_delete_safe_id_fix.zip

- 근로자 삭제 시 첫 번째는 실패하고 두 번째 삭제에서 반영되는 현상을 보강했다.
- 삭제 대상 ID 처리와 동기화 실패 대기열 처리 안정화를 목표로 했다.

### pc_30_shared_settings_sync_fix.zip

- 공휴일 자동갱신 인증키와 외부 API 인증키는 PC 환경값이 아니라 회사 운영 공용 설정값으로 보고 동기화 대상에 포함했다.
- 서버 주소와 백업 폴더 위치는 계속 PC별 환경값으로 제외했다.

### pc_31_holiday_auto_sync_apply_fix.zip

- 공휴일 인증키가 다른 PC로 동기화된 뒤 설정 화면과 공휴일 데이터 반영이 바로 이어지지 않는 문제를 보강했다.
- 동기화 후 공휴일 설정 재적용, 자동갱신 실행 트리거, 공휴일 캐시 초기화, 근태/급여 화면 반영을 목표로 했다.

### pc_32_worksite_business_select_fix.zip

- 사업장 신규 등록/수정 창에 `소속 사업자` 드롭다운을 추가했다.
- 사업자가 많지 않으므로 복잡한 트리 구조 대신 회사표 아래 등록/수정 영역에서 사업자를 고르는 방식으로 결정했다.
- 신규 사업장 등록 시 어느 사업자 소속인지 선택할 수 있게 했다.
- 기존 사업장 수정 시 다른 사업자로 이동할 수 있게 했다.
- 왼쪽에서 선택한 사업자가 있으면 기본값으로 미리 선택되도록 했다.
- 사업자를 선택하지 않은 상태에서도 등록 창에서 직접 소속 사업자를 고를 수 있게 했다.

### pc_33_common_refresh_after_save_fix.zip

- 사업자/사업장/근로자/차량 등 등록·수정·삭제 또는 동기화 후 일부 화면 목록이 바로 갱신되지 않는 문제를 공통 새로고침 방식으로 보강했다.
- 예: A사업자에 B사업장을 추가했는데 PC2 동기화 직후 안 보이고, C사업자 추가 같은 다른 저장 동작 후에야 B사업장이 표시되는 문제.
- `src/main_window.py`에서 데이터 변경 신호를 감지하고 숨겨진 관련 화면들의 새로고침을 예약하는 방향으로 수정했다.
- 현재 작업 중인 화면은 직접 흔들지 않고, 숨겨진 화면 위주로 갱신해 입력 중 화면 흔들림을 줄이는 방향이다.
- 저장 조건, 권한 조건, 동기화 판단, 계산식, 서버 API 조건은 건드리지 않았다.

## 현재 확인해야 할 테스트 목록

### PC_A / PC_B 한 컴퓨터 테스트

- 한 PC에서 두 프로그램을 테스트할 때는 폴더를 나눈다.
  - 예: `C:\green_test\PC_A`
  - 예: `C:\green_test\PC_B`
- 서버 주소는 같게 둔다.
- 로컬 데이터/DB는 달라야 한다.
- 백업 위치는 가능하면 다르게 둔다.
- PC_A에서 동기화 전 등록한 자료가 PC_B에 바로 보이면 같은 로컬 DB를 쓰는 상태라 테스트가 정확하지 않다.

### 사업자/사업장 동기화 테스트

1. PC1에서 A사업자에 B사업장 추가
2. PC1 동기화
3. PC2 동기화
4. PC2에서 다른 저장 동작 없이 A사업자 아래 B사업장이 바로 보이는지 확인
5. 사업장 수정 창에서 소속 사업자를 다른 사업자로 변경 가능 여부 확인
6. 변경 후 PC2와 모바일에 반영되는지 확인

### 근로자/차량/근태 동기화 테스트

- PC1에서 근로자 등록 → 동기화 → PC2에서 표시 확인
- PC2에서 근로자 등록 → 동기화 → PC1에서 표시 확인
- 모바일에서 근로자 등록 → PC1/PC2 동기화 후 표시 확인
- 모바일에서 근태 입력 → PC1/PC2 동기화 후 근태표 반영 확인
- PC1에서 차량 등록 → PC2 동기화 후 표시 확인
- PC2에서 차량 등록 → PC1 동기화 후 표시 확인

### 삭제 동기화 테스트

1. PC1에서 테스트 근로자 삭제
2. 오류 없이 바로 사라지는지 확인
3. PC1 동기화
4. 동기화 실패 표시가 없는지 확인
5. PC2 동기화
6. PC2에서도 삭제됐는지 확인
7. 모바일에서도 삭제됐는지 확인

### 공휴일 자동갱신 테스트

1. PC1에서 공휴일 자동갱신 인증키 입력
2. 자동갱신 사용 체크
3. 설정 저장
4. PC1 동기화
5. PC2 동기화
6. PC2 공휴일 설정에 인증키/사용 체크 반영 확인
7. 근태관리/급여관리 달력에서 공휴일 표시 확인
